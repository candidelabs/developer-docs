# Account Types

AbstractionKit ships several account classes. This page is the single reference for which one to use, what it is actually called, and how the import alias you see in the guides maps to the real class.

## Which one should I use?

- **New wallet** (you are creating a fresh smart account for the user): use `SafeMultiChainSigAccountV1`. It works single-chain and as the Safe Unified Account for multichain, so one class covers both.
- **Existing EOA** (the user already has an externally owned account and wants to upgrade it in place): use `Simple7702AccountV09`. It upgrades the existing EOA via EIP-7702 delegation, keeping the same address.

Everything below is the full list, including legacy classes.

## The `SafeAccount` alias

Most guides import the account class with a short alias for readability:

```ts
import { SafeMultiChainSigAccountV1 as SafeAccount } from "abstractionkit";
```

`SafeAccount` in a guide always refers to whichever class is named on the left of the `as` in that import. It is not a separate class. The default across these docs and the [abstractionkit-examples](https://github.com/candidelabs/abstractionkit-examples) repo is `SafeMultiChainSigAccountV1` (the Safe Unified Account), because one class covers both single-chain and multichain use.

When you read a snippet, check the import to see which real class `SafeAccount` points at on that page.

## Choosing a class

| Class | Use case | EntryPoint | External signer method |
|-------|----------|------------|------------------------|
| `SafeMultiChainSigAccountV1` | New wallets (recommended). A new smart account that works single-chain and as the Safe Unified Account for multichain. | v0.9 | `signUserOperationWithSigners(op, signers[], chainId)` for single chain, `signUserOperationsWithSigners(ops[], signers[])` for multichain |
| `SafeAccountV0_3_0` | Legacy v0.7 compatibility | v0.7 | `signUserOperationWithSigners(op, signers[], chainId)` |
| `SafeAccountV0_2_0` | Legacy v0.6 compatibility | v0.6 | `signUserOperationWithSigners(op, signers[], chainId)` |
| `Simple7702AccountV09` | Existing EOAs upgraded in place via EIP-7702 (recommended for that case) | v0.9 | `signUserOperationWithSigner(op, signer, chainId)` |
| `Simple7702Account` | EIP-7702 legacy v0.8, used in the v0.8 to v0.9 migration example | v0.8 | `signUserOperationWithSigner(op, signer, chainId)` |
| `Calibur7702Account` | EIP-7702 Calibur (passkeys, key management) | v0.8 (default) | `signUserOperationWithSigner(op, signer, chainId)` |

The "external signer method" column lists the signing call for the [external signer](/wallet/abstractionkit/external-signers/) path. The getting-started guides instead pass a raw private key with `signUserOperation` / `signUserOperations`, which is the same account using a different signing entry point.

## Module version is not the EntryPoint version

The Safe class names refer to the Safe 4337 **module** version, not the EntryPoint version. This trips up both people and agents, so read the EntryPoint column above as the authoritative mapping.

- `SafeAccountV0_3_0` is the "0.3.0 module" Safe and runs on **EntryPoint v0.7**, not v0.8.
- `SafeAccountV0_2_0` is the "0.2.0 module" Safe and runs on **EntryPoint v0.6**.
- There is no Safe class for EntryPoint v0.8. The SDK goes from the v0.7 Safe straight to `SafeMultiChainSigAccountV1` on **EntryPoint v0.9**.
